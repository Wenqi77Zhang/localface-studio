# ADR-001：原生推理为默认、ComfyUI 为可选的混合后端

状态：已接受  
日期：2026-07-22

## 背景

旧项目使用 ComfyUI 节点图完成生成、控制和评价。节点图便于理解实验逻辑，但把可交付产品直接绑定到 ComfyUI 和大量自定义节点，会增加版本漂移、打包、错误处理、自动化测试和许可证管理难度。

新版仍需要保留可视化思维，并为未来生成式编辑提供 ComfyUI 能力，同时第一版精准换脸必须可以独立、稳定运行。

## 决策

采用混合后端：

1. 默认后端是独立 Python 服务，通过 ONNX Runtime 执行检测、身份编码、换脸及可选增强。
2. Web 前端使用自己的节点式流程图展示真实任务阶段和事件。
3. ComfyUI 作为可选的进程外服务，通过 HTTP/WebSocket API 适配器通信。
4. 默认后端不导入、复制或修改 ComfyUI 内部模块。
5. 前端只依赖统一任务契约，不感知具体模型或工作流实现。
6. 视频扩展复用同一组推理端口，在外层增加帧源、人物跟踪和时间一致性组件。

## 核心接口边界

- `FaceDetector`：检测人脸并返回稳定的任务内检测 ID、边界框、关键点和置信度。
- `FaceIdentityEncoder`：在内存中生成临时身份表示。
- `FaceSwapper`：把来源身份应用到指定目标检测对象。
- `FaceEnhancer`：可选、可跳过且不得覆盖基础结果。
- `QualityEvaluator`：输出有明确含义和适用范围的检查结果。
- `WorkflowBackend`：统一原生后端和 ComfyUI 后端的任务事件、取消、结果与错误。
- `ArtifactStore`：管理任务临时文件和删除，不允许模型组件自行决定永久路径。

## 任务状态模型

任务至少支持：

- `queued`
- `validating`
- `detecting`
- `waiting_for_selection`
- `aligning`
- `swapping`
- `blending`
- `enhancing`
- `evaluating`
- `exporting`
- `completed`
- `failed`
- `cancelled`

状态转换由后端控制。前端流程图不得凭时间推测状态。

## 选择理由

- 原生 ONNX 路径适合单张照片精准换脸，资源占用和故障面较小。
- 进程外 ComfyUI 保留节点工作流价值，同时降低 GPL 组件与产品代码直接组合的许可证风险。
- 统一契约允许先用模拟后端测试产品，再接入真实模型。
- 模型适配器便于未来替换商业授权模型。
- 图片和视频可以共享推理组件，避免第二次重写核心逻辑。

## 代价

- 需要维护前端流程图和后端事件协议。
- 可选 ComfyUI 适配器会增加独立的集成测试。
- 原生后端与 ComfyUI 工作流可能产生不同结果，必须分别记录后端标识和版本。

## 许可证约束

ComfyUI 当前采用 GPL-3.0。阶段 8 只通过公开 API 与独立进程通信，不把 ComfyUI 打包进默认应用；任何联合分发方案都必须在实施前重新审查。此文件是工程风险控制记录，不构成法律意见。

## 被拒绝的方案

### 方案 A：所有功能都运行在 ComfyUI 中

拒绝原因：默认产品将受到 ComfyUI、自定义节点、工作流版本和模型目录结构的共同约束，自动化测试和错误隔离较差。

### 方案 B：完全移除 ComfyUI

拒绝原因：损失旧项目的可视化实验能力，也不利于未来接入 ControlNet、扩散编辑和复杂工作流。

### 方案 C：在同一 Python 进程直接导入 ComfyUI

拒绝原因：耦合和许可证边界更复杂，升级或崩溃会直接影响默认服务。

## 复审触发条件

- ComfyUI 许可证或 API 发生重大变化。
- 默认后端需要扩散模型能力。
- 产品需要把 ComfyUI 一并打包分发。
- 商业化或公网部署进入实施阶段。
