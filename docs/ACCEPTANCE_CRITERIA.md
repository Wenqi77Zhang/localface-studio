# LocalFace Studio 第一版验收标准

状态：阶段 0 基线  
日期：2026-07-22

## 1. 验收原则

- 每项必须给出可重复的测试步骤和证据。
- “在一张示例图上看起来不错”不等于通过。
- 不适用项必须说明理由，不能静默跳过。
- 自动指标不能替代人工质量检查或授权判断。
- 性能阈值在阶段 4 首次目标设备基准后冻结；阶段 0 不虚构速度目标。

## 3. 功能验收

### AC-F01 启动与健康检查

- 在目标 Windows 设备按文档启动成功。
- 默认访问地址是回环地址，不监听全部网卡。
- 前端能够展示后端、GPU 执行器、模型就绪状态和应用版本。
- 模型缺失时显示配置指导，不崩溃或假装就绪。

### AC-F02 上传与输入校验

- PNG、JPEG 和 WebP 的有效样例可以上传。
- 文件扩展名与内容不符、损坏、超限和无法解码的文件被拒绝。
- 未确认授权时不能提交图片任务。
- 拒绝信息说明原因和修复方法，不显示内部堆栈。

### AC-F03 来源人脸

- 单张清晰来源脸可以被检测。
- 无脸来源图被拒绝。
- 来源图出现多脸时不得静默选择错误人物；必须要求选择或提示更换图片。
- 来源脸过小或质量不足时给出明确警告或拒绝。

### AC-F04 目标多脸选择

- 目标图中的有效人脸均显示检测框和编号。
- 编号在同一任务的选择、确认和换脸阶段保持一致。
- 用户必须且只能选择一个目标检测 ID。
- 实际修改的人脸与用户确认的人脸一致。
- 未被选择的人脸不得被替换。

### AC-F05 换脸结果

- 输出宽高默认与目标图一致。
- 换脸区域外不存在大面积不可解释变化。
- 目标姿势、表情、发型、服装、背景和整体构图得到保持。
- 基础结果和增强结果是两个独立工件。
- 模型、GPU 或融合失败时任务为失败，不产生误导性“成功”文件。

### AC-F06 流程可视化

- 校验、检测、等待选择、对齐、换脸、融合、增强、评价和导出节点均存在。
- 节点状态来自后端任务事件。
- 失败节点显示错误类别，后续不适用节点显示跳过。
- 页面刷新后能够恢复当前任务状态，或明确说明任务已失效。

### AC-F07 下载与标记

- 用户可下载 PNG，并可选择 JPEG。
- 输出包含应用名、版本、AI 编辑声明、处理时间、后端标识和水印状态。
- 元数据不包含身份向量、原始绝对路径、用户名或设备唯一标识。
- 可见水印默认开启。
- 用户可以关闭水印，但关闭前看到清晰提示；AI 元数据仍然写入。

### AC-F08 取消与清理

- 排队和运行中的任务均有明确取消行为。
- 成功、失败、取消和超时路径都能进入清理流程。
- 应用异常退出后再次启动，会清理超过保留期的遗留任务。

## 4. 隐私与安全验收

### AC-S01 网络隔离

- 默认只绑定 `127.0.0.1`。
- 非允许 Host 或 Origin 的请求被拒绝。
- 未携带有效会话令牌的状态变更请求被拒绝。

### AC-S02 文件安全

- 用户原始文件名不会成为服务器路径。
- 路径穿越测试不能逃离任务目录。
- 超大像素、压缩炸弹、损坏图片和不支持的多帧输入被安全拒绝。

### AC-S03 敏感数据

- 人脸身份向量不写入磁盘、日志、API 响应或前端存储。
- 日志扫描不出现图片二进制和完整用户绝对路径。
- 默认不发送遥测或外部网络请求。

### AC-S04 模型供应链

- 只有模型清单登记且哈希匹配的文件可加载。
- 哈希不匹配时后端拒绝启动该模型。
- 受限模型首次启用前显示许可证并取得确认。
- 商业模式不能加载标记为非商业的模型。

### AC-S05 GitHub 安全

- 仓库扫描不包含模型、压缩权重、人脸图片、身份向量、任务缓存、生成结果或密钥。
- Git 历史在首次公开推送及每次发布前接受同样检查。
- 旧项目原始 PDF 不进入 Git；只提交已脱敏 Markdown 复盘。
- 公开仓库中的阶段进展日志不得包含本机绝对路径、访问令牌或人物身份信息。

## 5. 质量验收

测试集必须覆盖：

- 正面、侧面和不同俯仰角。
- 柔光、侧光、逆光和色温差异。
- 戴眼镜、刘海和部分遮挡。
- 小脸和高分辨率目标图。
- 单人和多人目标图。
- 不同性别表达、肤色和年龄段；素材必须具备相应用途许可。

报告至少包含：

- 检测成功率和失败原因分布。
- 对齐后身份相似度及其模型限制。
- 关键点/姿势保持检查。
- 换脸区域外变化检查。
- 基础结果与增强结果的配对比较。
- 匿名随机顺序的人工评分表。

不得使用完整生成图与来源证件照之间的全图 LPIPS 作为身份保持结论。

## 6. 性能与稳定性验收

- 在 RTX 5060 Laptop GPU、约 8 GB 显存和约 16 GB 内存的目标设备实测。
- 报告冷启动、模型加载、单次推理和导出的分项耗时。
- 报告峰值显存、进程内存和输出尺寸。
- 连续运行测试不能出现持续增长的显存或内存泄漏。
- GPU 不可用、显存不足和驱动不兼容均产生明确错误或已记录的 CPU 降级。
- 性能目标必须基于首次基准结果与用户体验共同冻结。

## 7. 可复现性验收

- 项目使用目录内独立 `.venv`。
- 前后端依赖版本和哈希可追踪。
- 新环境可按文档完成安装和启动。
- 模型来源、许可证、文件名、版本、大小和 SHA-256 有记录。
- 安装过程不依赖未记录的全局 Python、Node 或手工复制步骤。
- 自动化测试提供单条入口命令并生成清晰结果。

## 8. 架构扩展验收

- 没有 ComfyUI 时，默认图片换脸功能完整。
- ComfyUI 适配器使用进程外 API，不被默认服务直接导入。
- 前端只依赖统一任务契约。
- 存在可测试的 `WorkflowBackend`、`FaceDetector`、`FaceSwapper` 和 `FaceEnhancer` 边界。
- 视频扩展可以在不改变图片任务核心契约的前提下新增帧源和跟踪器。

## 9. 文档与交付验收

- 用户手册覆盖安装、启动、换脸、导出、清理和常见错误。
- 开发文档解释架构、数据流、模型适配器、任务状态和测试策略。
- 许可证页区分代码许可证与模型权重许可证。
- 测试素材清单包含来源、作者、许可证、下载日期和哈希。
- GitHub README 明确当前研究后端不得商业使用。
- 所有只能人工完成的操作提供逐步截图或逐条指令。

## 10. 第一版发布门禁

只有在以下项目全部满足后才允许标记 `v1.0.0`：

- 功能验收全部通过。
- 高风险隐私威胁均已缓解或由产品负责人书面接受。
- 许可证清单无未解释的默认依赖或默认模型。
- 目标设备性能和稳定性测试通过。
- 干净环境复现通过。
- GitHub 资产扫描通过。
- 产品负责人完成最终人工视觉验收。
