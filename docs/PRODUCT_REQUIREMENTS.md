# LocalFace Studio 产品需求文档

状态：已冻结，等待阶段 0 验收  
确认日期：2026-07-22  
产品负责人：琪琪  

## 1. 背景

旧项目基于 SDXL、ControlNet、IPAdapter、Florence-2 和 FaceDetailer，研究不同提示词对美学质量和身份保持的影响。旧报告具有探索价值，但存在单样本、混杂变量、身份评价指标不成立和工作流不可完整复现等问题。

新版不机械复刻旧工作流。第一版转为可实际演示和使用的传统精准换脸产品：只迁移来源人物的面部身份，尽量保持目标图的姿势、表情、发型、服装、背景、构图和光照。

## 2. 产品目标

在 Windows 11、NVIDIA GeForce RTX 5060 Laptop GPU（约 8 GB 显存）和约 16 GB 内存的目标设备上，提供一个稳定、可解释、可测试、完全本地运行的 Web 应用。

用户应能完成以下闭环：

1. 阅读并确认图片使用授权与模型许可提示。
2. 上传一张来源人脸图和一张目标照片。
3. 查看目标照片中检测到的全部人脸。
4. 每次选择一名目标人物。
5. 执行精准换脸并查看实时流程状态。
6. 比较输入、基础结果和可选增强结果。
7. 下载带 AI 编辑元数据的 PNG 或 JPEG。

## 3. 第一版范围

### 3.1 必须实现

- 本地 Web 界面。
- 来源图人脸检测。
- 目标图多脸检测、可视化编号与单人选择。
- 每次只替换一名被选中的目标人物。
- 原生 Python/ONNX 换脸后端。
- 节点式处理流程可视化和逐阶段状态。
- 输入校验、可理解的错误信息和任务取消。
- 结果前后对比与下载。
- AI 编辑元数据。
- 默认开启、允许关闭的可见水印。
- 临时文件自动清理和敏感日志抑制。
- GPU 执行器检测、显存释放和明确的 CPU 降级策略。
- 模型许可证提示和非商业后端使用门禁。
- 自动化测试和可重复安装文档。

### 3.2 第一版明确不做

- 视频换脸。
- 实时摄像头换脸。
- 公网部署、用户账户、付费或云端 GPU。
- 批量替换目标图中的多个人物。
- 未经许可的名人、陌生人或敏感场景素材。
- 依靠提示词改变背景、服装、姿势或画风。
- 把 ComfyUI 作为默认运行依赖。
- 分发许可证不明或限制商用的模型权重。

## 4. 用户与使用场景

### 4.1 第一阶段用户

- 项目负责人本人。
- 课程或作品集演示的现场观众。
- GitHub 上阅读和复现代码的开发者。

### 4.2 未来用户

- 获得照片主体授权的普通创作者。
- 在取得合规模型授权后，将后端集成到公开或商业产品的开发团队。

未来商业可能性不表示当前研究权重可以商业使用。

## 5. 功能需求

### FR-01 输入与授权

- 未勾选授权确认时不得创建任务。
- 支持常见 PNG、JPEG 和 WebP 输入；实际格式必须通过文件内容检测，不只依赖扩展名。
- 对文件大小、像素数量、通道数和解码异常设置限制。

### FR-02 人脸检测与选择

- 来源图没有人脸时拒绝执行。
- 来源图多脸时要求用户明确选择来源人脸，或在第一版界面中阻止歧义输入。
- 目标图显示全部有效检测框与稳定编号。
- 用户必须选中且只能选中一个目标人物。
- 结果必须记录所选检测 ID，而不是依赖不稳定的数组位置。

### FR-03 精准换脸

- 换脸只作用于所选人脸区域。
- 目标图输出尺寸默认保持不变。
- 未选人脸和人脸区域以外的内容不得出现不可解释的变化。
- 模型加载失败、执行器失败或检测失效时不得返回伪成功。

### FR-04 结果与增强

- 基础换脸结果必须独立保存于当前临时任务中。
- 人脸增强是可选后处理，不得覆盖基础结果。
- 用户可比较基础结果与增强结果。
- 下载前应用所选水印策略并写入 AI 编辑元数据。

### FR-05 流程可视化

- 显示：校验、检测、选择、对齐、换脸、融合、可选增强、质量检查、导出。
- 节点状态至少包括等待、运行、成功、跳过和失败。
- 前端展示的状态必须来源于后端事件，不得使用纯计时动画伪造进度。

### FR-06 数据生命周期

- 每个任务使用随机且不可预测的任务 ID 和独立临时目录。
- 会话关闭、任务取消或超过保留期后清理输入、中间文件和输出副本。
- 不持久化人脸身份向量。
- 日志不得包含图片字节、身份向量或用户文件的完整绝对路径。

## 6. 非功能需求

### NFR-01 隐私

默认仅绑定 `127.0.0.1`，不得在未认证情况下监听所有网卡。

### NFR-02 可复现性

依赖必须锁定版本；模型必须记录来源、版本、许可证、文件名和 SHA-256。安装说明不得依赖口头步骤。

### NFR-03 可替换性

检测、身份编码、换脸、增强、评价和工作流后端必须通过稳定接口隔离。前端不得直接依赖特定模型名称。

### NFR-04 可观察性

记录阶段名称、耗时、错误类型和执行器，但不记录敏感图片内容。性能目标在阶段 4 首次基准测试后以实测值冻结，不在阶段 0 凭空承诺秒数。

### NFR-05 资源约束

后端不得同时常驻多个不必要的大模型。任务完成和失败路径都必须释放可释放的 GPU/CPU 资源。内存和显存不足必须转化为可解释错误。

### NFR-06 可测试性

核心编排必须可以注入模拟后端，以便在没有模型或 GPU 的环境中测试完整产品流程。

## 7. 输出与元数据

- 默认输出格式为 PNG；JPEG 作为可选格式。
- 输出保留目标图像素尺寸，除非用户明确选择缩放。
- 元数据至少包含：应用名称、应用版本、处理时间、AI 编辑声明、后端标识、可见水印状态。
- 元数据不得包含人脸身份向量、原始完整路径、用户姓名或设备唯一标识。
- 第一版的元数据声明不等同于加密来源证明；若未来需要可验证来源链，应单独评估 C2PA。

## 8. 模型与商业边界

- 第一版允许使用仅限非商业研究的权重进行个人学习和演示。
- 应用必须在模型启用前显示限制并要求确认。
- 研究模型不得随应用安装包或 GitHub 仓库分发。
- 当前研究后端不得用于收费服务、商业部署或公司产品集成。
- 商业化前必须替换为已获商业授权或基于合规数据自行训练的模型，并重新完成许可证和隐私审查。

## 9. 未来扩展约束

- 视频后端复用相同的检测、身份、换脸和增强接口。
- 视频扩展新增帧解码、人物跟踪、时间一致性、音视频复用和中断恢复，不修改第一版图片任务契约。
- ComfyUI 作为独立进程，通过 API 适配器通信；默认后端不得导入 ComfyUI 内部模块。

## 10. 成功定义

第一版只有在功能、隐私、许可证、可重复安装、自动化测试和目标设备实测全部通过后才可称为完成。单张示例图片成功不构成产品完成。
